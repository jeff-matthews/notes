Goals

Drive customer success by:

  • Creating and publishing comprehensive AffiniPay web application user docs (non-admin).
  • Collaborating with support/training to ensure coverage of known user pain points.

Developer Portal Best Practices is a great resource that analyzes developer doc site trends. I think the article's insights will help us design our doc ecosystem.

Background and strategic fit

The only type of "help" docs that AffiniPay web app (aka virtual terminal [VT]) users currently have access to are support articles. The support articles are a great resource, but they're limited in scope and answer very specific questions about using the application.

We need help docs that walk first-time and returning users through major concepts and tasks in an organized format in addition to the knowledge base/support articles. This type of documentation will be crucial to customer success when we launch subscriptions/product billing because successful implementation will require users to configure products, businesses, and subscriptions in the web app UI.

Developers will be able to configure businesses and subscriptions programmatically via the Product Billing API, but they'll have to use the web app UI to configure products.

See AMP Docs Website for my initial proposals for incorporating subscriptions and future services into the developer doc site. Upon review, management declined both proposals for the short term delivery to Clio because of my work estimates, but I think we should revisit the subject. Since we're providing comprehensive product docs for the VT that includes subscriptions, I don't understand why we wouldn't add subscriptions to our existing developer doc set as well. I may not be able to make ALL the revisions I consider ideal to the developer docs to support subscriptions as outlined in the proposal, but it's still a better user experience than delivering a doc site identical to our current site but with subscription content only.

Assumptions

  • Although the impetus for "help" docs is the subscriptions/product billing feature, the scope of this effort should not be limited to subscriptions/product billing topics exclusively because we should provide docs for all web app functionality.
  • We'll host help docs on a subdomain like https://docs.afinipay.com.
  • We'll add admin docs as part of another epic.

1Scalable directory structure

As an AffiniPay tech writer, I want to configure the site directory to be scalable so that I can add content for new services/features in the future quickly without having to re-architect the site. A sound design meets the following criteria:

  • Ability to easily single source content via include statements between developer/api docs and help docs
  • Ability to centralize assets, such as JS, CSS, HTML templates, and images
  • Ability to add content for new services/features as individual repos to promote modularity and prevent potential merge conflicts and source control complexity in a multi-author scenario

2Landing pages

As a developer or manager/decision maker visiting https://developer.affinipay.com for the first time, I want the landing page to answer the following questions so that I can quickly assess how AffiniPay can help me or my company:

  • What type of documentation does AffiniPay provide?
  • Where can I learn how to use the VT web app?
  • What type of integrations does AffiniPay support?
  • What type of integration is right for me and where do I go to learn more about how to implement it?

We need the following landing pages to address these questions:

  1. Main - points to https://docs.affinipay.com and provides a path to developer/api docs or help docs
  2. Help - points to https://docs.affinipay.com/help and provides tiles for popular features (e.g., payments, refunds, reports)
  3. API - points to https://developers.affinipay.com and provides a path to third-party app integrations or payment forms
Must haveWill need to incorporate TP-153 changes
3Help docs logoAs a developer, manager/decision maker, or VT web app user, I want to be able to visually differentiate between the developer/api docs and help docs so that I can quickly identify where I am on the site. We need a new logo in the top-left corner of the page for the proposed Help section. See wireframes for an example. Create a new HTML page layout for help pages that contains this logo.Must haveMust be in SVG format and be the same size as the existing logo for developer/api docs
4Navbar links

As a developer, manager/decision maker, or VT web app user, no matter where I am on the site, I want to be able to easily navigate between API docs and Help docs so that I can refer to all types of documentation when searching for information.

The top navbar on all pages should contain 3 permanent links:

  1. Help - points to https://docs.affinipay.com and contains web app user docs
  2. API - points to https://developers.affinipay.com and contains developer-facing docs
  3. Contact - is a mailto link that points to devsupport@affinipay.com

When a user is exploring content on the "docs" subdomain, the Help link should be underlined or otherwise visually indicate that the user is in the Help section. When a user is exploring content on the "developers" subdomain, the API link should be underlined or otherwise visually indicate that the user is in the API section.

Must have
5Toggle TOC

As a developer, I want to be able to collapse the table of contents (TOC) on API reference pages so that I can take advantage of more screen real estate when viewing API reference information.

Add a hamburger menu button adjacent to the left nav TOC that vertically collapses and expands the TOC.

Nice to haveSee https://developer.gitbook.com for an example.
6Open source docs

As a developer, manager/decision maker, VT web app user, or AffiniPay staff member, I want to be able to contribute to documentation through Github pull requests so that I can help improve AffiniPay documentation.

Host all docs on Github and encourage contributions via pull requests (PR). Add an octacat logo to denote docs are open source and link to AffiniPay org repo.

Nice to haveWhile I don't expect a lot of feedback, I think this is ultimately beneficial because it allows customers and people within our organization to contribute doc updates without needing access to our private Bitbucket repo.
