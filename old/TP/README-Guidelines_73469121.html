Tech Pubs : README Guidelines

Created by Jeff Matthews, last modified on Apr 13, 2017

Github is an important platform for collaborative software development. It's also a channel for AffiniPay to share tools with our partner/developer community. We need to provide concise, accurate, and standardized documentation for all AffiniPay open source modules. Standardizing our READMEs makes creating and maintaining them easier, promotes a better developer experience, and helps us earn trust within the partner/developer community.

An effective README thoroughly explains:

  • What a module does (with context)
  • What a module looks like in action
  • How to install it
  • How to use it (with working examples)
  • All API methods/functions

Outline

The order in which we present information in a README is important because we care about people's time. Developers should be able to quickly scan our READMEs to determine whether a module meets their needs or if they need to move on to something else. All AffiniPay READMEs should contain, at minimum, the following sections (denotes optional sections):

  • Description
  • Overview (title optional)
  • Background
  • Installation
  • Configuration
  • Usage
  • API
  • Support
  • Contribute
  • License

Use the make-readme Yeoman generator to quickly scaffold an AffiniPay standards-compliant README from the command line.


Description

Use the repo's Description field in Github to add a one-line description that briefly summarizes what the module does.

A modern combo-box built for React and Bootstrap

Overview

This section doesn't require a title in the actual README. It refers to the first paragraph(s) before the Usage section. This is where you explain what the module does in greater detail with use cases and other context that clarifies why someone might want to use the module. More detail of the module's routines or methods, lengthy code examples, or other in-depth material should be given in subsequent sections.

Background

Consider including a Background section if your module depends on important but not widely known abstractions or other ecosystems. This is also a great place to explain the module's motivation if similar modules already exist.

Installation

If a developer has read this far down, then they're probably sold on trying out the module. Don't forget to include system prerequisites. Even simple npm install module instructions belong in this section.

<module> is written in Node.js, which must be installed prior to use. Node.js includes the Node Package Manager (NPM), which is used for installing dependencies.

After cloning the repository, install dependencies and build:

$ npm install

After building, you can install the tool globally to run it from anywhere:

$ npm install --global

Configuration

If your module requires configuration before developers can use it, explain it in this section.

  1. Contact AffiniPay Support to create a partner OAuth2 application.
  2. Configure OAuth and Redirect URI
  3. Retrieve your AffiniPay Partner Application Client ID and Secret

Usage

Show developers what a module looks like in action so they can quickly determine whether the example meets their needs. If it does, then they can read on for further details. This section should contain clear, runnable code examples.

$ virtsecgroup --help

  Usage: index [options] [input-file]

  Options:

    -h, --help                  output usage information
    --dump-ast                  Dump parse AST
    --dump-charts-on-error      Dump parse charts if an error occurs
    -o, --output <output-file>  Specify the Terraform output filename
    -v, --verbose               Increase verbosity of output


API

The name, description, and usage of this module all sound appealing to me. I'm very likely to use this module at this point. I just need to scan the API to make sure it does exactly what I need and that it will integrate easily into my codebase. The API section should detail the module's objects and functions, their signatures, return types, callbacks, and events in detail. Types should be included where they aren't obvious. Caveats should be made clear.

Support

Encouraging developers to report issues/feature requests and actively addressing their feedback makes the project better and facilitates trust between AffiniPay and the developer community.

Please use Github Issues for bug reports or feature requests.

Contribute

Providing guidance on how to contribute to the project will help both AffiniPay and non-AffiniPay developers work together efficiently.

Contributions in the form of GitHub pull requests are welcome. Please adhere to the following guidelines:

  • Before embarking on a significant change, please create an issue to discuss the proposed change and ensure that it is likely to be merged.
  • Follow the coding conventions used throughout the project, including 2-space indentation and no unnecessary semicolons. Many conventions are enforced using eslint.
  • Any contributions must be licensed under the ISC license.

License

Most READMEs show this at the bottom, but it's better to show it at the top if you're using a non-permissive license (e.g., GPL). Developers are likely to reject a module very quickly if it has a license incompatible with their work. AffiniPay generally sticks to the permissive MIT/ISC/BSD flavors, but consult your manager if you're unsure. One known example of using a non-permissive license is the WooCommerce plugin. We release it under the GPL license, which is either a WordPress or WooCommerce requirement.

Best Practices

  • If you talk about other modules, ideas, specialized terminology, or people, make that reference text a link so that visitors can more easily grok your module and the ideas it builds on. In markdown, you can also keep footnotes at the bottom of your document, so referring to them several times throughout becomes cheap.
  • Include information on types of arguments and return parameters if it's not obvious. Prefer convention wherever possible (cb probably means callback function, num probably means a Number, etc.).
  • Include the example code from the Usage section as a file in your repo--maybe as example.js. It's great to have README code that users can actually run if they clone the repository.
  • Be judicious in your use of badges. They add visual noise to your README and generally only function if the user is reading your Markdown in a browser online, since the images are often hosted elsewhere on the internet.
  • If your module provides a CLI (command line interface) instead of (or in addition to) a programmatic API, show usage examples as command invocations and their output. If you create or modify a file, cat it to demonstrate the change before and after.
  • API formatting is highly bikesheddable. Use whatever format you think is clearest, but make sure your format expresses important subtleties:
    • Which parameters are optional, and their defaults.
    • Type information where it is not obvious from convention.
    • Don't shy away from providing a tiny example of an API function's use if it is not obvious or fully covered in the Usage section. However, this can also be a strong signal that the function is too complex and needs to be refactored, broken into smaller functions, or removed altogether.

Checklist

  • A helpful checklist to gauge how your README is coming along:
  • One-liner explaining the purpose of the module
  • Necessary background context & links
  • Potentially unfamiliar terms link to informative sources
  • Clear, runnable code examples
  • Installation instructions (including prerequisites)
  • Extensive API documentation
  • Caveats and limitations mentioned up-front
  • Doesn't rely on images to relay critical information
  • License
