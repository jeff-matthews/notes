The developer landing page represents an important opportunity for us to make a good impression on prospective and current developers; an opportunity that we're not fully leveraging. Comprehensive documentation that's easy to navigate is paramount to fostering a perception of a well-supported product/platform, which helps drive more wide-spread adoption among developers.

Our current developer landing page design fails to promote all of the other documentation resources we offer, including API tutorials, API reference, and code libraries. We must provide a better experience for developers visiting the site, especially when comparing ourselves to competitors. Prospective and current developers should be able to quickly find the same resources they've come to expect from other platforms on the AffiniPay landing page.

This report summarizes popular design patterns across developer landing pages for competing platforms and provides recommendations for the AffiniPay developer landing page based on an analysis of these patterns.

Common Design Patterns

All landing pages sampled in this report share similar design patterns, notwithstanding a few minor exceptions, in terms of layout, information architecture, and functionality. 

Structural components

Only three pages employ the sidebar component (e.g., Stripe, Invoiced, Chargify) at the developer landing page level. The remaining five pages employ a header + body + footer layout and reserve the additional sidebar component for documentation subpages.

This table summarizes structural components across pages and denotes the distribution of pages using certain component elements:

Background graphic

Five of the landing pages sampled in this report include an "above-the-fold" background graphic with a tag line in the body component. See the screenshots in the following Content types section for examples.

Content types

Eight pages provided links to the following content types (predominantly in the body component):

  • Getting started
  • Integration methods
  • API tutorials
  • API reference
  • Code samples & libraries

Links are represented by cards, tiles, graphics, or lists in the body component; usually about half-way down the page, but "above-the-fold."

Search

Six of the pages sampled in this report include a search box in the header component. Search functionality varies from robust (type-ahead with results dynamically populating a drop-down [Stripe]) to basic (enter a search term, press enter, and view a separate page of results [Adyen]) to nonexistent (Zuora).

  • Robust - 3
  • Basic - 3
  • None - 3

Sign in/Sign up

Six of the landing pages sampled in this report include a Sign in button (or some variation thereof) in the header component. It's unclear how this might impact site functionality in all cases. For example, does signing in provide interactive API support or does it simply take you to the platform "dashboard" for performing administrative tasks? More research is required.

Help & support

Seven of the landing pages sampled in this report include links to support resources in the header component. Resources range from a contact form or mailto: link for support inquiries to a link to a separate page of options including contact information and forum and knowledgebase access.

Recommendations

It would be beneficial to incorporate some of the common design patterns discussed in this report into the AffiniPay developer landing page. The following sections provide guidance on some preliminary design requirements.

Structural components

Use header + body + footer components and reserve the sidebar component for documentation subpages.

This table recommends which elements to use in each component:

ComponentDescription
Header
  • Logo
  • Support resources (drop-down w/ links)
  • Sign in/sign up
  • Site search
SidebarFull developer site content navigation. Only accessible from site subpages.
Body
  • Above-the-fold background graphic with tag line
  • Choose an integration type
  • Cards with graphics that link to content type subpages
Footer

About + Ts&Cs/Disclaimer + Copyright only

Background graphic

We're currently using a background graphic that has a parallax effect when scrolling. There's just not enough content on the page to require scrolling, so the effect is lost.

We should still endeavor to guide the user experience by asking them to choose an integration path before we show cards for other content types, like we currently do and like Adyen does, but I think we should provide some sort of interactive diagram that shows developers what each integration method entails when they click something. For example, if I clicked a "payment form" icon, only elements pertinent to that method come to the forefront and everything else drops to the background. This preserves the context for all other methods and helps show how methods are related.

Another idea is to create an interactive experience based on a binary decision tree. For example, a user clicks "Help me decide which integration is right for me" and we present them with two options. After they click an option, we can present them with another set of options until we've determined the best integration path based on their inputs. From there, we can drop them on the diagram with an integration path already selected so they can see a system overview.

Content types

No changes to our current content types are necessary. We already provide all the resources developers expect. The purpose of the landing page redesign is to make these resources more visible.

We should add cards for the following content types in the body component:

  • Getting started
  • Integration methods (interactive diagram)
  • API tutorials
  • API reference
  • Code samples & libraries

We may need to create a pseudo landing page for the API tutorials since there are multiple pages associated with tutorials. We can link directly to a single page for all other content types.

Search

Our current search solution (lunr.js) is sufficient for now. Eventually, it'd be nice to have a more robust search solution with a type-ahead feature like Stripe, but that's either expensive (Swiftype) or requires time for custom development.

Sign in/Sign up

Ideally, a sign-in button would authenticate a developer, retrieve their API keys, and redirect them to the doc site where they could try out the API through an interactive console. In the absence of that functionality, we can either:

  1. Redirect to a new page explaining the VT dashboard, other product flavors (e.g., lawpay, cpacharge), and provide a link to sign up if they don't have an account.
  2. Redirect to https://secure.affinipay.com. I'm not sure there's a lot of value in providing a sign-in link in the latter case, but it seems to be a common design pattern among competitors.

Help & support

Add a drop-down menu in the header component with the following links:

  • Dev support email
  • FAQ
  • Support knowledgebase

This would also be a good place for a future troubleshooting section that explains common issues and how to work around them.
