Background and strategic fit

Leadership's marketing strategy for releasing new products and product APIs (e.g., billing, subscriptions, invoicing) is unclear. The primary purpose of this planning document is to outline doc-centered options based on the assumption that leadership adopts AMP as a new brand for product marketing purposes. 

As a contingency, this document outlines the requirements for extending the existing developer site to accommodate new "AMP" APIs.


As we release new /amp-api features, we'll need to build a new site to support the AMP brand:

  • Marketing site: educates all audience types about the AMP platform and drives traffic to the developer site
  • Developer site: educates technical audiences about our APIs and how to use them

Assumptions

  • New products and APIs will be released under the AMP brand
  • The AffiniPay developer doc site (https://developers.affinipay.com) will enter "maintenance mode"
  • Marketing will design, build, test, and own the AMP marketing site that caters to all audiences and promotes platform features
  • Development will design, build, test, and own the AMP developer site--a subdomain of the AMP marketing site--that caters to technical users

Rationale

This is an exploratory exercise to further discuss with management. Leadership has not yet decided on how to brand AMP features. Until that determination is made, legacy and new docs may end up living on the same site. That's problematic for the following reasons:

  • Different API base URLs may cause confusion
  • Legacy site doesn't support swagger/standardized definition as an input, so we'll either have to refactor the site to accept the input and build definitions for all APIs or manually move (i.e., copy/paste) AMP API reference from the swagger definition to source files in the legacy site repo
  • We continue ignoring the wider implications of bad info architecture. We should have a coordinated effort to build an AMP-specific landing page where we can market the platform (not just payments) and promote new/cool features to a wider audience of users and then funnel them into more technical info (e.g., API reference, developer tutorials)
  • If we stick with the legacy site, we need to redesign the landing page and expand the focus of introductory topics to include non-payment gateway APIs
